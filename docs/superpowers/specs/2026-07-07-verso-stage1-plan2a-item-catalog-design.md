# Verso Stage 1 ‚Äî Plan 2a: the item-analysis bundle & catalog

**Date:** 2026-07-07
**Status:** Approved design, ready for implementation planning.
**Parents:** [Stage 1 blocks design](2026-07-07-verso-stage1-blocks-design.md) (¬ß5 catalog, ¬ß7 coherence risk) ¬∑ Plan 1 (block engine, merged `01b186d`)

---

## 1. Decisions

1. **The bundle (chosen over per-block calls and hybrid).** Item-scope analysis is generated by ONE hidden block, `item-analysis`: a single holistic LLM call per item (simpleyt's proven approach), returning structured JSON. The visible analysis blocks are **pure extractions** of the bundle ‚Äî zero LLM calls. Rationale: full cross-section coherence, ~1/8th the cost, and the prerequisite system makes derivation free to express. Trade-off accepted: one larger prompt to maintain; regenerating any section regenerates the bundle.
2. **Citations: numbered markers + refs map (chosen over inline fact-ids and structured claims).** The bundle prompt receives the document's facts numbered `1..N`; sections carry `[n]` markers; a refs map (`n ‚Üí { factId, url }`) is embedded in the bundle JSON. Same idiom as the existing `briefRefs` ‚Äî one citation style product-wide. Extractions inherit citations through the content (markers travel with text) and copy the refs map into their own `citations` ‚Äî which resolves the "derivations can't inherit citations" deferral for this family without touching the resolver.

## 2. The bundle block

- `id: 'item-analysis'`, `name` user-invisible, `scope: 'item'`, `hidden: true`, `staleness: 'on-demand'`.
- Prerequisites: `raw-content` + `item-metadata` + **`item-facts`** (new primitive, ¬ß4).
- Generator: one `client.complete(prompt, { jsonSchema })` call. Prompt = simpleyt's analysis prompt modernized: metadata + capped content + numbered facts; output schema with one field per section (executive_summary, key_themes, detailed_breakdown, arguments_evidence, notable_quotes, strengths_weaknesses, actionable_takeaways, open_questions) + `refs: [{ n, factId?, url }]`.
- Cached output `content` = the JSON string; `citations` = the refs mapped to `{ factId?, url }[]`.

## 3. The extractions (visible catalog)

Each: `scope: 'item'`, prerequisite `[{ kind: 'block', blockId: 'item-analysis' }]`, `generate()` parses the bundle JSON, returns its section as markdown + the refs map as citations. **No LLM.**

| Block id | Section |
|---|---|
| `exec-summary` *(re-pointed ‚Äî prompt/generator replaced by extraction; id, name, tests' declaration shape updated)* | executive_summary |
| `key-themes` | key_themes |
| `detailed-breakdown` | detailed_breakdown |
| `arguments-evidence` | arguments_evidence |
| `notable-quotes` | notable_quotes (verbatim quotes + commentary) |
| `strengths-weaknesses` | strengths_weaknesses |
| `actionable-takeaways` | actionable_takeaways |
| `open-questions` | open_questions |

`tldr` unchanged (LLM condensation of exec-summary's content; its input is now the extracted section text ‚Äî same contract).

DAG proof: `item-analysis ‚Üí exec-summary ‚Üí tldr` (bundle ‚Üí free extraction ‚Üí tiny LLM derivation).

## 4. Engine touches (all of them)

1. `BlockInput` += `{ kind: 'item-facts' }`: the target document's facts. Resolver case (requires item target; loads via new loader; fingerprint = content hash over fact ids); store loader (facts where `documentId = target`, dossier-scoped); registry validation (item-capable blocks only).
2. `BlockDef` += `hidden?: boolean`; blocks route library listing filters `hidden`.
3. Nothing else changes: jobs, runner, staleness, routes, schema are untouched (bundle output fits the existing `block_outputs.content` text column).

## 5. Out of scope

- Page-scope variants (map-reduce Key Themes etc.) ‚Äî Plan 2b, together with fact-pool loader bounds + `facts.dossier_id` index.
- Wrapping brief ("Since you last looked") and Chronology as blocks ‚Äî Plan 2b.
- `auto-on-refresh` execution ‚Äî Plan 2b.
- Feed block and all UI ‚Äî Plan 3.

## 6. Acceptance

- Ancestor test, strengthened: one YouTube item in a dossier yields all eight simpleyt sections, each cited down to fact passages, for ~one LLM call (plus tldr's small one).
- Attaching a second/fifth/eighth analysis block to an already-analyzed item is instant (extractions hit the cached bundle).
- Refresh ‚Üí stale marking ‚Üí re-run with unchanged inputs re-verifies (no regeneration) per Plan 1 semantics.

## 7. Risks

- **Bundle JSON robustness:** LLM may emit imperfect JSON despite jsonSchema; extraction of a missing section must fail-soft (recorded as failed for that block, batch continues ‚Äî existing runner semantics).
- **Prompt size:** transcript cap (24k chars) + facts; watch token limits on long videos ‚Äî reuse the existing CONTENT_CAP pattern.
- **Section quality drift vs dedicated prompts:** accepted (the coherence/cost trade was decided consciously); revisit per-section if dogfooding shows a weak section.

## 8. Post-implementation notes (final review, 2026-07-07)

- **Pre-2a instances:** dossiers whose extraction-family instances predate 2a lack an `item-analysis` instance; runs record misses until the block is re-POSTed (idempotent attach re-runs the auto-attach loop). Decision: **accept the self-heal, no backfill** ‚Äî dev DB only, and the single affected dossier already has its bundle from the smoke run.
- **Deferred to Plan 3 (UI):** GET blocks returns hidden instances/outputs unfiltered (~18KB bundle JSON in the payload ‚Äî UI must filter, or move filtering server-side); POST allows attaching a hidden block by id (add a `def.hidden` guard when the library UI lands). Also: cap the numbered-facts list in the bundle prompt if per-document fact counts grow; consider memoizing `cachedOutput` reads per run if extraction counts rise.
